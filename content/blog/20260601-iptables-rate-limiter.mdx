[前回](/blog/20260529-distributed-token-bucket-design) は Python でアプリケーション層 (L7) のレートリミットを実装しました。今回はレイヤーを下げて、Linux カーネルに組み込まれた iptables を使った トランスポート層 (L4) での実装を試してみます。

## 実装

Docker コンテナ 2 つで検証します。

- attacker コンテナ: `ab` コマンドで大量リクエストを投げる攻撃者役
- victim コンテナ: nginx を立ててリクエストを受ける

iptables の `hashlimit` モジュールを使うと、送信元 IP ごとにレートリミットを設定できます。前回と同じパラメータ（`capacity=10, refill_rate=5/sec`）で書くと次のようになります。

```bash
# 確立済み接続・ループバックは無条件で通す
iptables -A INPUT -m conntrack --ctstate ESTABLISHED,RELATED -j ACCEPT
iptables -A INPUT -i lo -j ACCEPT

# TCP :80 の新規接続 (SYN) を IP ごとに 5 req/sec・バースト 10 に制限
iptables -A INPUT \
  -p tcp --dport 80 --syn \
  -m hashlimit --hashlimit-mode srcip --hashlimit-name http_limit \
  --hashlimit-above 5/sec --hashlimit-burst 10 \
  -j DROP
```

`--syn` は TCP 接続の最初の 1 パケット（SYN）だけを対象にするオプションです。新規接続 1 本あたり、トークンが 1 個消費されます。

## 比較

`ab -n 100 -c 10`（100 リクエスト、10 並列）を制限あり・なしの両方に投げました。

| 指標                | 意味                                                  | 制限なし     | 制限あり (5/sec) |
| ------------------- | ----------------------------------------------------- | ------------ | ---------------- |
| Time taken          | 100 件全部完了するまでの合計時間                      | 0.007 秒     | 18.4 秒          |
| Requests per second | 1 秒あたりに処理できたリクエスト数                    | 14,376 req/s | 5.44 req/s       |
| Connect: mean       | TCP 接続確立（SYN → SYN-ACK → ACK）にかかった平均時間 | 0 ms         | 1,481 ms         |

Requests per second は 5.44 で `--hashlimit-above 5/sec` で設定したレートを超えない値になっています。

Time taken が 18.4 秒になった理由は、最初の 10 件はバーストで即座に通りますが、その後は 5/sec のペースでしか通せないためです。
「残り 90 件 ÷ 5 件/秒 = 18 秒」 となり、計算上の予測とほぼ一致しています。
なお `Failed requests: 0` のままなのは、DROP されても TCP が再送して最終的には通るためで、失敗ではなく遅延として現れたということです。

**余談: 指数バックオフ**

`-j DROP` はパケットを黙って捨てるだけなので、クライアントは「返事が来ない」としか分かりません。[RFC 6298](https://www.rfc-editor.org/rfc/rfc6298.html#section-5) の規定により、TCP は再送のたびに待ち時間を 2 倍にします。

> The host MUST set RTO &lt;- RTO \* 2 ("back off the timer").

```text
1 秒待つ → 再送 → DROP
2 秒待つ → 再送 → DROP
4 秒待つ → 再送 → バケツが補充されていれば通る
```

再送待ちが積み重なるほど接続確立時間が伸びます。victim 側のカウンタを見ると、100 リクエストを送ったにもかかわらず DROP は 160 回マッチしていました。100 を超えているのは、DROP されるたびに TCP が SYN を再送するためだと思われます。

`Connect: mean` は TCP 接続確立（SYN → SYN-ACK → ACK）にかかった平均時間です。
SYN が DROP されると TCP は約 1 秒後に再送するため、1 回でも再送が発生すれば 1,000ms 以上かかります。1,481ms はそのあたりが積み重なった結果だと思います。

## 感想

Token Bucket はアプリのコードに組み込む必要がありましたが、iptables は 3 行追加するだけでアプリの外側からレートを制御できて便利でした。
ただし、L4 での制御はリクエストの内容を見ないため、API のエンドポイントごとに制限したい場合はアプリ層で実装する必要がありそうですね。
